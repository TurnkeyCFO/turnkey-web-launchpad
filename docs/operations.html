The GTM path is one continuous system.

The goal is not more disconnected automations. The goal is a single premium workflow that moves cleanly end to end.

Shared architecture

One core system, two brand adapters.

The same operating backbone should serve Turnkey CFO and Turnkey Web while allowing each brand to keep its own offers, tone, and delivery standards.

Core layers

  • Intake layer: leads, requests, and tasks from forms, social, outbound, email, and manual entry.
  • Qualification layer: fit, complexity, budget, urgency, and async compatibility.
  • Workflow layer: estimate, onboarding, delivery, approval, retention.
  • Integration layer: Sheets, Drive, Gmail, Slack, Telegram, GitHub, OpenRouter, n8n.

Canonical objects

  • lead, client, opportunity, estimate, onboarding packet, task, draft
  • approval, publish/send job, support request, report
  • Required state fields: owner, brand, source, stage, priority, fit score, complexity score, due date, approval status, failure state, retry count, last touched, next action

Testing and troubleshooting

Every workflow needs a happy path and a recovery path.

What should always be tested?

Happy path, missing data, approval rejection, connector failure, and retry or replay.

What does troubleshooting need?

Clear symptom, likely cause, check list, recovery step, and log entry for the failure.

What does good logging look like?

Owner, brand, stage, failure state, retry count, and next action should be visible without digging.
