Operating objective

Build a clear, agentic business system with strict fit, simple navigation, and measurable milestones.

This page is the working plan. It is meant to read like an operating memo, not a sales page: what we are building, why it matters, what gets priority, what gets deferred, how we test it, and how we know the system is healthy.

Strategic bias Fewer clients, better fit

Higher margin, lower complexity, more repeatable delivery.

Execution layer Codex + OpenClaw

The cockpit for planning, drafting, review, and troubleshooting.

Operating model Agent-first workflows

Standard states, clear approvals, simple handoffs, visible failures.

Primary goal Repeatable GTM path

Attract, qualify, convert, onboard, deliver, retain.

Operating principles

These are the rules that should shape every process decision.

Principle Meaning Practical rule
Automation is the product advantage The system should create leverage, not just save time. If a workflow cannot be standardized, it should be simplified or declined.
Clarity beats cleverness The process must be easy to understand at a glance. Use simple states, short labels, and consistent handoffs.
Premium means fit A smaller number of aligned clients is preferable. Filter out high-chaos, low-margin, heavily custom work.
One core, two brands Turnkey CFO and Turnkey Web should share the same backbone. Keep brand differences in content and rules, not in the whole system design.

Shared architecture

Build one reusable operating stack that can support both businesses without branching into two unrelated systems.

Layer Purpose Notes
Intake Capture leads, requests, tasks, and follow-ups. Sources: forms, email, Slack, social, outbound, manual entry.
Qualification Score fit, budget, complexity, and urgency. This is where low-fit work should be filtered out early.
Workflow engine Move work through estimate, onboarding, delivery, and retention. Each stage should have a clear owner and next action.
Integration layer Connect systems and keep state in sync. Sheets, Drive, Gmail, Slack, Telegram, GitHub, OpenRouter, n8n.
Visibility layer Show status, failures, approvals, and queue depth. This is what keeps the system manageable without guesswork.

Full GTM path

This is the standard path from lead to retained client. Every stage should be trackable and measurable.

01 Attract
Goal: capture interest in one intake stream.

Social, outbound, referrals, and site activity should all feed the same lead intake layer.

Success point Lead captured.

02 Qualify
Goal: decide whether the lead fits the operating model.

Score for fit, complexity, budget, timeline, and how well the client can work asynchronously.

Success point Qualified lead accepted.

03 Convert
Goal: move the opportunity into an approved engagement.

Estimates, proposals, and approvals should flow through a clear agent-driven sequence.

Success point Estimate issued, proposal approved.

04 Onboard
Goal: collect assets and expectations without chaos.

Requests for access, examples, goals, and constraints should be handled as a structured checklist.

Success point Onboarding complete.

05 Deliver
Goal: produce work through visible draft and approval states.

Research, outreach, tasks, drafts, reviews, approvals, and launches should all be auditable.

Success point First deliverable accepted, launch completed.

06 Retain
Goal: turn delivery into recurring value.

Reporting, follow-up, maintenance, and upsell or referral opportunities should be surfaced automatically.

Success point Recurring work retained, referral or upsell triggered.

Implementation roadmap

A four-week plan with a practical focus on architecture, workflow definition, testing, and launch hardening.

Week Focus Deliverables Exit criteria
Week 1 Architecture lock Shared core, canonical states, fit filter, current-system inventory. One agreed operating model that covers both brands.
Week 2 Workflow formalization Lead-to-estimate path, estimate-to-onboarding path, shared status language. The first fully documented workflow loop exists end to end.
Week 3 Testing and troubleshooting Happy-path tests, failure tests, retry paths, recovery playbooks. Known failure modes can be diagnosed and recovered cleanly.
Week 4 Launch hardening Turnkey Web launch path, repeatable CFO operating loop, weekly review cadence. The system can run with fewer, better-fit clients at higher quality.

Testing and troubleshooting

Every workflow should be tested the same way so the system is predictable and easy to support.

Test area What to verify Expected result
Workflow path Happy path, missing data, approval rejection, connector failure, retry. The system continues without losing ownership or state.
Logging Owner, brand, stage, failure state, retry count, next action. The status of any work item can be understood quickly.
Recovery Connector outage, rejected draft, timeout, replay. A documented next step exists for each failure type.
End-to-end Lead capture through retention for CFO and Web. The full GTM path is usable without manual rescue.

Decision rules

  • Prefer standard packages over custom one-off work.
  • Decline clients that require constant manual coordination.
  • Use agents for repeatable actions, not for everything.
  • Keep states, approvals, and ownership visible at all times.

Business focus

  • Turnkey CFO: stabilize outreach, revenue ops, admin, and visibility.
  • Turnkey Web: productize the delivery engine and launch path.
  • Shared core: one operating system across both brands.

Current success measures

  • Lead intake is consistent.
  • Qualification is strict.
  • Approvals are explicit.
  • Failures are diagnosable.
  • Milestones are visible.

Launch criteria

The system is ready when the following are true.

  • There is one shared architecture for both businesses.
  • The fit filter is defined and used consistently.
  • The GTM path is documented stage by stage.
  • The roadmap has clear weekly exit criteria.
  • The test plan covers happy path, failure, and recovery.
  • The system stays navigable without adding unnecessary complexity.
